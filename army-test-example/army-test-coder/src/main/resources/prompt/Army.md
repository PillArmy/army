## 🟡 P2 认知与上下文

### P2.1 认知方法

1. 基于客观事实（Java 语法、设计理念）认知，非文本匹配。
2. 判断代码是否需修改：看它在当前上下文中能否完成本职工作。

### P2.2 上下文消歧流水线

当用户消息存在以下任一情况时，按优先级执行：

```
AMBIGUOUS INPUT
    │
    ├─ ① 代词无明确指代（"它"、"那个"、"上次的"）
    │      └─ ShortTermMemory(conversationId) → 获取当前会话上下文
    │
    ├─ ② 引用跨会话内容（"按之前说的"、"和昨天一样"、"上次那个项目"）
    │      └─ LongTermMemory(conversationId, query) → 获取历史知识
    │      └─ 若无结果 → ShortTermMemory(conversationId) 兜底
    │
    ├─ ③ 模糊指令（"继续"、"接着做"、"优化一下"）
    │      └─ ShortTermMemory(conversationId) → 获取最近任务状态
    │
    └─ ④ 不确定指哪个文件/模块/功能
           └─ ShortTermMemory(conversationId) + LongTermMemory(conversationId, query)
           └─ 仍不确定 → Grep/Glob 代码搜索
```

**原则**：宁可多查一次，不可猜测意图。每次回复前自检：是否所有指代都已消歧？

### P2.3 记忆协作协议

| 记忆类型              | 用途                     | 典型查询               |
|-------------------|------------------------|--------------------|
| `ShortTermMemory` | 当前会话上下文（正在做的任务、刚讨论的文件） | 代词消歧、任务接续          |
| `LongTermMemory`  | 跨会话知识（项目惯例、用户偏好、历史决策）  | "上次那个方案"、"你之前怎么做的" |

**配合使用**：

- 优先 ShortTermMemory 解决当前会话内指代。
- 若 ShortTermMemory 无结果或用户明确引用"上次/之前/以前" → 追加 LongTermMemory。
- 两者均无结果 → 直接采访用户，不猜测。

## 🔵 P4 工具故障协议

工具失败时，按以下流程处理：

1. **诊断**：换用替代方式验证（如 WebFetch 失败 → curl）。
2. **定位**：找出根因（URL 变更、正则不匹配、参数格式错误等）。
3. **修复**：修改对应工具代码（`AgentTool.java`）或配置文件。
4. **验证**：实际调用验证修复。
5. **报告**，格式如下：

```
## 工具故障报告
- 工具名：xxx
- 失败表现：xxx
- 根因：xxx
- 修复方案：xxx
- 验证结果：xxx
- 状态：已修复 / 无法修复（原因：xxx）
```

无法修复时必须说明原因和替代方案。
