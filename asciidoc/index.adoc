== Preface

=== Introduction

Army is a better Java SQL framework offering the following core capabilities:

* A better way to write SQL in Java
* A better blocking ORM framework
* A higher-level database driver

=== Design Philosophy

1. **Don't create a new world ÔÇö just map the real world.** Respect database standards and dialect differences.
2. **We need standards, and we need dialects.** That's how the real world works.
3. **Keep it simple ÔÇö blocking by default, straightforward and predictable.**

=== Project Conventions

* Interfaces or classes whose names begin with an underscore (`_`) are **private to the Army framework** and should not be directly depended upon in user code.

=== Core Concepts at a Glance

Using Army follows a clear pipeline:

```
Domain classes (annotated)
Ôćĺ Annotation processor generates metamodel classes at compile time
Ôćĺ Criteria API builds SQL statements
Ôćĺ Dialect parser generates native SQL
Ôćĺ Session API executes statements
Ôćĺ Executor (StmtExecutor) drives JDBC execution
```

* **Domain Class** ÔÇö A POJO annotated with `@Table`, `@Column`, etc., describing the mapping between Java objects and database tables.
* **Metamodel Class** ÔÇö A `_`-suffixed class auto-generated at compile time by `ArmyMetaModelDomainProcessor`, providing type-safe field references.
* **Criteria API** ÔÇö A Java DSL for building SQL statements (SELECT, INSERT, UPDATE, DELETE).
* **Session API** ÔÇö A unified entry point for executing statements and managing transactions.
* **Dialect Parser** ÔÇö Translates standard Criteria into native SQL for each database.
* **Executor** ÔÇö `io.army.executor.StmtExecutor`, responsible for JDBC statement execution, parameter binding, and result set processing.

=== Your First Domain Class

Map a Java class to a database table using annotations:

[source,java]
----
@Table(name = "stock",
       indexes = @Index(name = "uni_stock_exchange_code", fieldList = {"exchange", "code"}, unique = true),
       comment = "stock")
public class Stock {

    @Generator(value = "io.army.generator.snowflake.Snowflake8Generator",
               params = @Param(name = "startTime", value = "1779012232202"))
    @Column
    public long id;

    @Column(notNull = true, updatable = false, precision = 5, comment = "exchange code")
    public String exchange;

    @Column(notNull = true, updatable = false, precision = 15, comment = "stock code")
    public String code;

    @Column(notNull = true, precision = 130, comment = "company name")
    public String name;

    @Column(notNull = true, precision = 10, defaultValue = "'NORMAL'", comment = "listing status")
    public StockStatus status;

    @Column(precision = 10, scale = 2, defaultValue = "0.00", comment = "offer price")
    public BigDecimal offerPrice;

    @Column(defaultValue = "DATE '1970-01-01'", comment = "listing date")
    public LocalDate listingDate;

    @Column
    public LocalDateTime createTime;

    @Column
    public LocalDateTime updateTime;

    @Column(comment = "version")
    public int version;
}
----

=== Compile-Time Metamodel Generation

`ArmyMetaModelDomainProcessor` automatically processes `@Table`-annotated classes at compile time and generates a static metamodel class `Stock_`:

[source,java]
----
@Generated(value = "io.army.modelgen.ArmyMetaModelDomainProcessor",
        date = "2024-01-02 07:13:54.605524+08:00",
        comments = "stock")
public abstract class Stock_ {

    public static final TableMeta<Stock> T;

    static {
        T =  _TableMetaFactory.getTableMeta(Stock.class);
    }

    // Field name constants (type-safe references)
    public static final String NAME = "name";
    public static final String OFFER_PRICE = "offerPrice";
    public static final String EXCHANGE = "exchange";
    public static final String CODE = "code";
    // ... other fields

    // Field metadata
    public static final FieldMeta<Stock> name = T.getField(NAME);
    public static final FieldMeta<Stock> offerPrice = T.getField(OFFER_PRICE);
    public static final PrimaryFieldMeta<Stock> id = T.id();
    // ... other field metadata
}
----

NOTE: Metamodel classes are the key to building **type-safe** Criteria statements.
All field references go through metamodel constants, eliminating string-based typos.

=== Creating a SessionFactory

==== Blocking Mode

[source,java]
----
public void createSessionFactory() {
    final Map<String, Object> envMap = new HashMap<>();
    envMap.put(ArmyKey.DATABASE.name, Database.MySQL);
    envMap.put(ArmyKey.DIALECT.name, MySQLDialect.MySQL80);

    final DataSource dataSource = createDataSource();
    sessionFactory = SyncFactoryBuilder.builder()
            .name("army-test")
            .packagesToScan(Collections.singletonList("io.army.example.stock.domain"))
            .datasource(dataSource)
            .environment(StandardEnvironment.from(envMap))
            .build();
}
----

==== What's Required vs. Optional

*FactoryBuilder* methods come with clear markers in the source code.
Here's the breakdown:

[cols="1,2,5a",options="header"]
|===
| Method | Status | Notes

| `name(String)`
| **Required**
| SessionFactory name, must be non-empty.

| `datasource(Object)`
| **Required**
| JDBC `DataSource` instance. Can be a `ReadWriteSplittingDataSource` for read/write splitting.

| `packagesToScan(List)`
| **Required**
| List of packages where domain classes (annotated with `@Table`) reside.

| `environment(ArmyEnvironment)`
| **Required**
| Environment configuration passed via `StandardEnvironment.from(map)`. Must contain at least `DATABASE` and `DIALECT` (see below).

| `classLoader(ClassLoader)`
| Optional
| Default: `Thread.currentThread().getContextClassLoader()`

| `loadStaticModel(boolean)`
| Optional
| Default `false`. Enable to load static metamodel classes after `TableMeta` creation for extra safety.

| `validateOnStartup(boolean)`
| Optional
| Default `false`. Enable for stronger consistency validation at startup.

| `jsonCodec(JsonCodec)` / `xmlCodec(XmlCodec)`
| Optional
| Custom JSON/XML codec.

| `fieldGeneratorFactory(FieldGeneratorFactory)`
| Optional
| Custom field generator factory.

|===

*ArmyKey* environment entries ÔÇö only two are truly mandatory, all others have sensible defaults:

[cols="1,1,5a",options="header"]
|===
| ArmyKey | Required? | Notes

| `DATABASE`
| **Yes**
| `Database.MySQL` / `Database.PostgreSQL` / `Database.SQLite`. No default ÔÇö framework needs to know which database to operate against.

| `DIALECT`
| **Yes**
| `MySQLDialect.MySQL80` / `PostgreDialect.POSTGRE15` / `SQLiteDialect.SQLite34`. No default ÔÇö must match your database version.

| `ZONE_OFFSET`
| No
| Timezone offset. When omitted, the framework uses database defaults.

| `DDL_MODE`
| No
| Default: `DdlMode.UPDATE` (create tables if absent, update columns if changed).

| `VISIBLE_MODE`
| No
| Default: `AllowMode.NEVER` (disable soft-delete visible field).

| `SQL_LOG_MODE`
| No
| Default: `SqlLogMode.OFF`. Set to `SqlLogMode.BEAUTIFY_DEBUG` during development.

| All other keys
| No
| Default values are defined in `ArmyKey` / `SyncKey`. See the source for the complete list.

|===

That means the minimal environment setup looks like the example above ÔÇö just `DATABASE` and `DIALECT`.
In the test example at `FactoryUtils.java`, extra keys like `SQL_LOG_MODE` and `VISIBLE_MODE` are set for development convenience, but they are not required for production.

TIP: If you forget `DATABASE` or `DIALECT`, the framework throws `IllegalStateException` with a clear message ÔÇö no silent failures.

== Domain Model Definition

Domain model definition is the starting point of Army.
This chapter walks you through how to use annotations to tell the framework: "_this Java class maps to that database table, this field maps to that column, this field needs a Snowflake ID generator..._" ÔÇö all in plain Java, no XML.

Behind the scenes, Army's `++TableMetaUtils++` and `++FieldMetaUtils++` read these annotations at runtime to build `++TableMeta++` objects, which are the immutable metadata blueprints that drive SQL generation.
The entry point is `++DefaultTableMeta#getTableMeta(Class, MetaContext)++`, which reads `++@Table++`, walks the class hierarchy, creates `++FieldMeta++` instances via `++TableFieldMeta#createFieldMeta++`, and caches the result ÔÇö so each domain class is parsed only once.

=== Annotation Overview

Here's a quick reference of all Army annotations.
Don't worry about memorizing them now ÔÇö each is explained in detail with real examples below.

[cols="1,3,1",options="header"]
|===
| Annotation | Purpose | Target
| `@Table` | Maps a class to a database table (name, comment, indexes) | Class
| `@Column` | Maps a field to a database column (name, type, nullability, default) | Field
| `@Generator` | Specifies how a field's value is generated (e.g., Snowflake ID) | Field
| `@GeneratorType` | When generation happens: `PRECEDE` / `POST` / `RUNTIME` / `DEFAULT` | Enum
| `@Param` | A name-value pair passed to a `FieldGenerator` | Annotation member
| `@FieldParam` | Binds a field name to parameter overrides (inside `@OverrideParams`) | Annotation member
| `@OverrideParams` | Overrides generator parameters of inherited fields | Class
| `@Index` | Defines a database index (name, fields, uniqueness, type) | Class (inside `@Table`)
| `@IndexField` | Per-column index configuration (collation, opclass, sort) | Class (inside `@Index`)
| `@SortOrder` | Index column sort: `ASC` / `DESC` / `DEFAULT` | Enum
| `@NullsOrder` | Index column NULL ordering: `NULLS_FIRST` / `NULLS_LAST` / `DEFAULT` | Enum
| `@Mapping` | Custom JavaÔćöSQL type mapping | Field
| `@Inheritance` | Marks the root of a single-table inheritance hierarchy | Class
| `@DiscriminatorValue` | Discriminator enum value for a child class in inheritance | Class
| `@MappedSuperclass` | Marks a base class whose fields are inherited by subclasses | Class
| `@DdlMode` | Controls DDL generation: `CREATE` / `NONE` / `DEFAULT` | Enum
| `@Codec` | Marker for future custom encoding/decoding | Field
|===

NOTE: Many annotation attributes support **placeholders** (`+${DEFAULT}+`, `+${RUNTIME}+`, `+${OPTIONAL}+`, `+${DEFAULT_VALUE}+`) that are resolved at runtime from the configuration file `META-INF/army/TableMeta.properties` on the classpath.
See <<TableMeta.properties ÔÇö Runtime Configuration>> for details.

=== @Table ÔÇö Table Mapping

`@Table` is placed on a Java class to tell Army: "_this class maps to a database table._" It controls the table name, index definitions, DDL behavior, and more.
At runtime, `++TableMetaUtils.tableName++` and `++TableMetaUtils.tableComment++` resolve its values; `++TableMetaUtils.schemaMeta++` handles `catalog`/`schema`, and `++TableMetaUtils.tableCreateDdl++` determines whether DDL is generated.

[TIP]
====
Both `name` and `comment` are **required** attributes.
If you omit them (or use empty strings), the framework throws `MetaException` at startup.
This is intentional ÔÇö Army refuses to guess silently.
====

==== Attributes

* `name` (required) ÔÇö The database table name.
Must not be in camelCase.
Supports placeholders: `++${DEFAULT}++` auto-converts the class simple name to snake_case (e.g., `StockQuotes` Ôćĺ `"stock_quotes"`); `++${RUNTIME}++` requires a property entry.
* `comment` (required) ÔÇö Table comment for DDL.
Supports `++${DEFAULT}++` (auto-derive from class simple name) and `++${RUNTIME}++`.
* `catalog` (optional) ÔÇö Catalog name, defaults to the database default.
Property key: `{className}.Table.catalog`.
* `schema` (optional) ÔÇö Schema name, defaults to the user's default schema.
Property key: `{className}.Table.schema`.
* `indexes` (optional) ÔÇö Array of `@Index` definitions.
If no index covers the `id` field, a unique primary key index is **created automatically**.
* `immutable` (optional, default `false`) ÔÇö When `true`, all `UPDATE` DML on this table is disallowed.
Parent tables with `@Inheritance` cannot be marked as immutable ÔÇö Army detects this at startup and throws `MetaException`.
* `allColumnNotNull` (optional, default `false`) ÔÇö When `true`, every column gets `NOT NULL` in DDL.
Note: reserved fields (`id`, `createTime`, `updateTime`, `version`, `visible`), discriminator fields, and primitive-typed fields are always `NOT NULL` regardless of this setting.
* `ddlMode` (optional, default `CREATE`) ÔÇö Controls DDL generation.
See `@DdlMode`.
* `tableOptions` (optional) ÔÇö Raw SQL fragment appended to `CREATE TABLE`, e.g., `"ENGINE=InnoDB DEFAULT CHARSET=utf8mb4"`.
* `partitionOptions` (optional) ÔÇö SQL partition clause, e.g., `"PARTITION BY RANGE (YEAR(create_time))"`.

==== Examples from stock-data

[source,java]
----
// Stock: table with a unique composite index
@Table(name = "stock",
       indexes = @Index(name = "${DEFAULT_VALUE}", unique = true,
                        fieldList = {"exchange", "code"}), // auto-generates: uni_stock_exchange_code
       comment = "stock")
public class Stock { }

// StockQuotes: table with unique index on (stockId, date)
@Table(name = "stock_quotes",
       indexes = @Index(name = "${DEFAULT_VALUE}", unique = true,
                        fieldList = {"stockId", "date"}), // auto-generates: uni_stock_quotes_stockId_date
       comment = "daily stock quotes")
public class StockQuotes { }

// UploadRecord: table with multiple non-unique indexes
@Table(name = "upload_record",
       indexes = {
           @Index(name = "${DEFAULT_VALUE}", fieldList = "userId"),
           @Index(name = "${DEFAULT_VALUE}", fieldList = "fileHash")
       },
       comment = "uploaded document record")
public class UploadRecord { }
----

TIP: Use `++${DEFAULT_VALUE}++` as the index name to let the framework auto-generate `+uni_{table}_{col}+` or `+idx_{table}_{col}+`.
Use `++${DEFAULT}++` only when you want to override the generated name via `META-INF/army/TableMeta.properties`.
For most cases, `++${DEFAULT_VALUE}++` is the right choice.

=== @Column ÔÇö Column Mapping

`@Column` maps a Java field to a database column. **Without `@Column`, a field is just a plain Java field** ÔÇö not persisted, not included in any SQL, not part of the table schema.
The framework's `++TableMetaUtils.columnName++` resolves the column name, `++TableMetaUtils.columnPrecision/columnScale++` resolve numeric attributes, and `++FieldMetaUtils.columnComment++` resolves the comment text.

==== Built-in Conventions (Zero-Config)

Army provides smart defaults for many `@Column` attributes ÔÇö you only need to specify what differs from the convention:

[cols="2,3,3",options="header"]
|===
| Attribute | Convention | How It Works
| `name` (not set) | `camelCase` Ôćĺ `snake_case` | `offerPrice` Ôćĺ `"offer_price"`, `stockId` Ôćĺ `"stock_id"`
| `comment` (not set) | Auto-generated for reserved fields | `id`Ôćĺ`"primary key"`, `createTime`Ôćĺ`"create time"`, `updateTime`Ôćĺ`"update time"`, `version`Ôćĺ`"version that's update counter of row"`, `visible`Ôćĺ`"visible for soft delete"`
| `precision` (not set) | By Java type | `String`Ôćĺ`255`, `BigDecimal`Ôćĺ`16`, `Enum`Ôćĺmax label length + 5, `UUID`Ôćĺ`36`, `Vector`Ôćĺ`1024`
| `scale` (not set) | By Java type | `BigDecimal`Ôćĺ`3`, temporal types (`LocalDateTime`/`OffsetDateTime`/`LocalTime`/`OffsetTime`)Ôćĺ`6` (microsecond precision)
| `defaultValue` (not set) | By Java type | ŠĽ░ňÇ╝š▒╗ň×őÔćĺ`"0"`, `String`Ôćĺ`"''"`, `LocalTime`Ôćĺ`"'00:00:00'"`, `LocalDate`Ôćĺ`"'1970-01-01'"`, `LocalDateTime`Ôćĺ`"'1970-01-01 00:00:00'"`
| `notNull` (not set) | `false` ÔÇö except for reserved fields | `id`/`createTime`/`updateTime`/`version`/`visible` and discriminator fields are always `NOT NULL`
| `collation` (not set) | Empty ÔÇö no collation clause | Set explicitly only when needed, e.g., `"en_US.utf8"`
|===

[IMPORTANT]
====
For business fields (non-reserved), if you don't provide a `comment`, Army throws `MetaException`.
Reserved fields get automatic comments, but your business columns must be documented.
====

==== Reserved Fields

Army recognizes five field names as **reserved** ÔÇö they are auto-managed by the framework and have special behavior:

[cols="1,1,2,2",options="header"]
|===
| Field Name | `NOT NULL` | INSERT Behavior | UPDATE Behavior
| `id` | Always | Value from `@Generator` | **Never updatable**
| `createTime` | Always | Auto-set to current time | **Never updatable**
| `updateTime` | Always | Auto-set to current time | **Always updatable**, auto-refreshed on every UPDATE
| `version` | Always | Auto-set to `0` | **Always updatable**, auto-incremented (optimistic lock)
| `visible` | Always | Auto-set to `true` | **Always updatable** (soft delete toggle)
|===

[TIP]
====
Reserved fields **never need `notNull`** ÔÇö Army enforces it automatically.
Reserved fields (except `id`) **must not have `@Generator`** ÔÇö Army throws `MetaException` if you try.
====

==== Attributes

* `name` (optional) ÔÇö Column name.
If empty, camelCaseÔćĺsnake_case.
Supports `++${DEFAULT}++` and `++${RUNTIME}++`.
Property key: `{className}.{fieldName}.Column.name`.
* `comment` (optional) ÔÇö Column comment for DDL.
Reserved fields auto-generate.
Enum fields with `++${DEFAULT}++` auto-generate `"enum, @see {EnumClass}"`.
Business fields without a value throw `MetaException`.
Supports `++${DEFAULT}++` and `++${RUNTIME}++`.
* `notNull` (optional, default `false`) ÔÇö Declares `NOT NULL`.
Forced `true` when: the field is reserved, a discriminator, primitive-typed, or `Table.allColumnNotNull` is `true`.
* `insertable` (optional, default `true`) ÔÇö Whether included in `INSERT`.
Overridden for reserved fields: `createTime`/`updateTime`/`version`/`visible` are always insertable; fields with `PRECEDE` generators are always insertable; `id` with `POST` generator is insertable only on child tables.
* `updatable` (optional, default `true`) ÔÇö Whether included in `UPDATE`.
Overridden: `id`/`createTime`/discriminator are **never** updatable; `updateTime`/`version`/`visible` are **always** updatable; fields on `immutable` tables are never updatable.
* `precision` (optional, default `-1`) ÔÇö Total digit count or length.
Defaults by type (see table above).
Use `Column.DEFAULT_EXP` (`-8150`) to resolve from `META-INF/army/TableMeta.properties`.
* `scale` (optional, default `-1`) ÔÇö Fractional digit count.
Defaults by type (see table above).
Use `Column.DEFAULT_EXP` to resolve from properties.
* `defaultValue` (optional) ÔÇö SQL expression for column default. **String values must include quotes**: `"'NORMAL'"` Ôćĺ `DEFAULT 'NORMAL'`.
Date values use SQL syntax: `"DATE '1970-01-01'"`.
Supports `++${DEFAULT}++` for type-appropriate zero values.
* `collation` (optional) ÔÇö Column collation, e.g., `"en_US.utf8"`.
Property key: `{className}.{fieldName}.Column.collation`.

==== Override via TableMeta.properties

The following `@Column` attributes support placeholder expressions.
Place the file at `META-INF/army/TableMeta.properties` on the classpath:

.`META-INF/army/TableMeta.properties`
[source,properties]
----

# Override column name
# Used when @Column(name = "${DEFAULT}") or @Column(name = "${RUNTIME}")

com.example.domain.Stock.name.Column.precision=512

# Override column scale (e.g., increase decimal places to 4)
com.example.domain.Stock.offerPrice.Column.scale=4

# Override column default value (SQL expression)
com.example.domain.Stock.status.Column.defaultValue='ACTIVE'

----

=== @Generator / @GeneratorType ÔÇö Field Value Generation

`@Generator` controls **how a field's value is produced** ÔÇö before INSERT (application-side, like Snowflake ID) or after INSERT (database auto-increment). `@GeneratorType` is the enum that tells the framework *when* to generate.

At runtime, `++TableFieldMeta++`'s constructor reads `@Generator` and `@GeneratorType` to create `++GeneratorMeta++` instances.
If the type is `RUNTIME` or `DEFAULT`, it calls `++FieldMetaUtils.loadGeneratorStrategy++` to look up `GeneratorStrategy` from `META-INF/army/TableMeta.properties`.

==== Generator Types

[cols="1,4",options="header"]
|===
| Type | When to Use
| `PRECEDE` (default) | Application generates the value **before** INSERT. Use for Snowflake IDs, UUIDs, or custom ID generation. The value from your generator is written into the `INSERT` statement.
| `POST` | Database **auto-increment** column. No code runs ÔÇö the DB assigns the value during INSERT, and Army reads it back afterward. **Only supported on the `id` field**, and the Java type must be `int`, `long`, `Integer`, `Long`, `String`, or `BigInteger`.
| `RUNTIME` | Defer the decision to `META-INF/army/TableMeta.properties`. Army reads the key `{className}.{fieldName}.Generator.type` and loads a `GeneratorStrategy`. Property value format: `POST`, `PRECEDE`, or `DEFAULT`.
| `DEFAULT` | Let the framework decide. For child tables in an `@Inheritance` hierarchy, defaults to `POST`; otherwise, behaves like `PRECEDE`.
|===

[IMPORTANT]
====
Discriminator fields (the field named in `@Inheritance`) and reserved fields except `id` (`createTime`, `updateTime`, `version`, `visible`) **must not** have `@Generator` ÔÇö Army throws `MetaException` at startup.
====

==== Override via TableMeta.properties

When `@Generator(type = GeneratorType.RUNTIME)` is used, the actual generator strategy is loaded from `META-INF/army/TableMeta.properties`.
Two keys are supported:

.`META-INF/army/TableMeta.properties`
[source,properties]
----
# File: META-INF/army/TableMeta.properties

# Option 1: Switch generator type at runtime (POST, PRECEDE, or DEFAULT)
com.example.domain.Stock.id.Generator.type=POST

# Option 2: Load a custom GeneratorStrategy (takes precedence over .Generator.type)
# Format: fully.qualified.StrategyClass[:jsonParams]
com.example.domain.Stock.id.GeneratorStrategy=io.army.generator.Snowflake8GeneratorStrategy:{"startTime":1779111192831}

# Override SQL type mapping for the id column (optional)
com.example.domain.Stock.id.Mapping.value=io.army.mapping.SqlBigIntType
----

IMPORTANT: The `GeneratorStrategy` value contains a colon (`:`) separating the class name from optional JSON parameters.
The JSON is parsed by the strategy's static `create(String)` factory method.

==== Attributes

* `type` (optional, default `PRECEDE`) ÔÇö The `GeneratorType` enum value.
* `value` (optional) ÔÇö Fully-qualified class name of the `FieldGenerator` implementation. **Required** for `PRECEDE`.
Ignored for `POST` (since `POST` uses the database, not application code).
* `params` (optional) ÔÇö Array of `@Param` name-value pairs passed to the generator at initialization.

==== Examples from stock-data

Each entity uses the **Snowflake-8 generator** with a unique `startTime` to produce non-overlapping ID ranges across tables:

[source,java]
----
// Stock entity: startTime=1779012232202
@Generator(value = "io.army.generator.snowflake.Snowflake8Generator",
           params = @Param(name = "startTime", value = "1779012232202"))
@Column
public long id;

// StockQuotes entity: different startTime for a different ID space
@Generator(value = "io.army.generator.snowflake.Snowflake8Generator",
           params = @Param(name = "startTime", value = "1779111192831"))
@Column
public long id;
----

TIP: Each table should use a **different `startTime`** so Snowflake produces non-colliding ID ranges.
This prevents primary key conflicts when joining data across tables.

=== @Param / @FieldParam / @OverrideParams ÔÇö Generator Parameters

These three annotations work together to pass and override generator configuration. `++FieldMetaUtils.getOrcreateFieldParmMap++` creates the parameter map; `++FieldMetaUtils.overrideParamMapIfNeed++` applies `@OverrideParams` overrides.

`@Param` ÔÇö A simple **name-value pair** passed to a generator as `@Generator(params = ...)`.
The generator parses the string value into its needed type:

[source,java]
----
// Snowflake8Generator reads "startTime" as a long
@Generator(value = "io.army.generator.snowflake.Snowflake8Generator",
           params = @Param(name = "startTime", value = "1779012232202"))
@Column
public long id;
----

`@FieldParam` ÔÇö Binds a field name to replacement parameters inside `@OverrideParams`:

[source,java]
----
@FieldParam(name = "id", params = @Param(name = "startTime", value = "1600000000000"))
----

`@OverrideParams` ÔÇö Class-level annotation for subclasses to change generator parameters of inherited fields **without rewriting the `@Generator` annotation**:

[source,java]
----
@OverrideParams(fields = {
    @FieldParam(name = "id", params = @Param(name = "startTime", value = "1600000000000")),
    @FieldParam(name = "uid", params = @Param(name = "startTime", value = "1600000000000"))
})
public class ChildEntity extends ParentEntity { }
----

TIP: `@OverrideParams` is the recommended way to give subclasses **different Snowflake ID spaces** without redeclaring `@Generator`.

=== @Mapping ÔÇö Custom Type Mapping

`@Mapping` overrides Army's default JavaÔćöSQL type inference.
Armey's `++_MappingFactory++` automatically maps common types (String, Long, BigDecimal, LocalDateTime, etc.) to sensible SQL types.
You only need `@Mapping` when:

* Storing binary data in a text column with a specific charset
* Using database-specific types like MySQL `SET` or PostgreSQL `ENUM`
* Mapping collection types (List, Set) to JSON/JSONB
* Overriding the framework's type choice (e.g., forcing `CHAR` for a hash)

==== Attributes

* `value` (optional) ÔÇö Fully-qualified class name of the `MappingType` implementation.
* `type` (optional) ÔÇö `Class` reference of the `MappingType`. **Takes precedence over `value`**.
* `params` (optional) ÔÇö Extra parameters for the mapping type, e.g., the enum type name for PostgreSQL `ENUM`.
* `func` (optional) ÔÇö Java method reference for constructing/creating instances of the mapped type.
Format: `"ClassName::new"` (constructor) or `"ClassName::methodName"` (static factory).
Consumed by `PgSingleRangeType.fromMethod`, `PgMultiRangeType.fromMethod`, and their array variants via `PgRangeType.createRangeFunction`.
Resolved by `_ArmyPgRangeType.loadConstructor` / `loadFactoryMethod`.
* `charset` (optional) ÔÇö Character set name. **Required** when `value` is a `TextMappingType` representing binary data (e.g., `InputStream`, `Path`).
* `elements` (optional) ÔÇö Element type(s) for collection mappings.
Required for `ElementMappingType` implementations like MySQL `SET`.
Not needed for array types ÔÇö the component type is obtained via reflection.

==== Examples from stock-data

[source,java]
----
// SHA-256 hash Ôćĺ fixed-length CHAR(44) instead of VARCHAR
@Column(precision = 44, comment = "file hash (SHA-256, standard base64)")
@Mapping("io.army.mapping.SqlCharType")
public String fileHash;

// List<Long> Ôćĺ JSONB (PostgreSQL) or JSON (MySQL), defaulting to '[]'
@Column(notNull = true, defaultValue = "'[]'", comment = "vector ID list")
@Mapping("io.army.mapping.PreferredJsonbType")
private List<Long> vectorIdList;
----

=== @Index ÔÇö Index Definitions

`@Index` defines a database index within `@Table#indexes`.
At runtime, `++TableMetaUtils.createIndexList++` reads all `@Index` annotations, resolves index names and types, and builds `IndexMeta` objects.
If no index covers the `id` field, a unique primary key index is appended automatically.

==== Attributes

* `name` (required) ÔÇö Index name.
Supports all placeholders.
See the naming convention below.
* `fieldList` (optional) ÔÇö Simple list of Java field names.
Each name refers to a field in the domain class and is resolved to its column name at runtime.
* `fields` (optional) ÔÇö `@IndexField[]` with per-column settings (collation, opclass, sort order). **Takes precedence over `fieldList`** when both are specified.
* `unique` (optional, default `false`) ÔÇö Whether the index enforces uniqueness.
The primary key index is always unique.
* `type` (optional) ÔÇö Index access method.
Supports `BTREE`, `HASH`, `FULLTEXT`, `SPATIAL` (MySQL); `btree`, `hash`, `gist`, `gin`, `brin`, `hnsw` (PostgreSQL).
Empty string (default) = database default.
Supports `++${DEFAULT}++`, `++${RUNTIME}++`, `++${OPTIONAL}++` placeholders.

==== Index Name Convention

[cols="2,4",options="header"]
|===
| Placeholder | Generated Name
| `++${DEFAULT_VALUE}++` | `+uni_{table}_{col1}_{col2}+` (unique) or `+idx_{table}_{col1}_{col2}+` (non-unique)
| `++${DEFAULT}++` | Same as above, but consults `META-INF/army/TableMeta.properties` first; falls back to `DEFAULT_VALUE` behavior
| Literal string | Used as-is
|===

==== Examples from stock-data

[source,java]
----
// Stock: unique composite index ÔÇö no duplicate (exchange, code) pairs
@Table(name = "stock",
       indexes = @Index(name = "${DEFAULT_VALUE}", unique = true,
                        fieldList = {"exchange", "code"})) // Ôćĺ uni_stock_exchange_code

// StockQuotes: unique index ÔÇö one quote record per stock per trading day
@Table(name = "stock_quotes",
       indexes = @Index(name = "${DEFAULT_VALUE}", unique = true,
                        fieldList = {"stockId", "date"})) // Ôćĺ uni_stock_quotes_stockId_date

// UploadRecord: two non-unique indexes for fast lookup
@Table(name = "upload_record",
       indexes = {
           @Index(name = "${DEFAULT_VALUE}", fieldList = "userId"),  // Ôćĺ idx_upload_record_userId
           @Index(name = "${DEFAULT_VALUE}", fieldList = "fileHash") // Ôćĺ idx_upload_record_fileHash
       })
----

==== @IndexField ÔÇö Per-Column Index Configuration

For scenarios requiring per-column settings (collation, opclass, sort order), use `@IndexField` instead of simple `fieldList`:

[source,java]
----
// Standard B-tree index with column collation
@Index(name = "${DEFAULT_VALUE}", unique = true, type = "btree",
    fields = {
        @IndexField(name = "exchange"),
        @IndexField(name = "code", collation = "en_US.utf8")
    })

// PostgreSQL HNSW vector index for embedding similarity search
@Index(name = "${DEFAULT_VALUE}", type = "hnsw",
    fields = @IndexField(name = "embedding", opclass = "vector_cosine_ops"))
----

[cols="1,3",options="header"]
|===
| Attribute | Description
| `name` (required) | Java field name of the indexed column
| `collation` (optional) | Column collation, e.g., `"en_US.utf8"`. PostgreSQL only. Supports `++${DEFAULT}++` and `++${OPTIONAL}++`
| `opclass` (optional) | Operator class for PostgreSQL, e.g., `"vector_cosine_ops"`, `"jsonb_path_ops"`. Supports `++${DEFAULT}++` and `++${OPTIONAL}++`
| `order` (optional) | Sort order: `SortOrder.ASC` / `DESC` / `DEFAULT`
| `nulls` (optional) | NULL ordering: `NullsOrder.NULLS_FIRST` / `NULLS_LAST` / `DEFAULT`
|===

=== @MappedSuperclass ÔÇö The Base Domain Pattern

`@MappedSuperclass` marks a class whose `@Column`-annotated fields are **inherited by subclasses**, but which does **not** map to any database table.
It's purely a field-template mechanism.

At startup, `++TableMetaUtils.mappedClassPair++` walks the class hierarchy upward from the domain class: it collects every superclass with `@MappedSuperclass` or `@Table` (reversing the list so the topmost superclass comes first), then `++createFieldMetaList++` collects all `@Column` fields from each class in order.

Unlike `@Inheritance`, a `@MappedSuperclass` has no discriminator column and cannot be queried directly.

==== The Inheritance Hierarchy

The `army-example-stock-data` module uses a three-level `@MappedSuperclass` chain:

```
MinBaseDomain<T>           Ôćĺ @Column createTime         (auto-set on INSERT, never updatable)
  ÔööÔöÇ BaseDomain<T>         Ôćĺ + @Column updateTime, version  (auto-set on INSERT/UPDATE; version auto-incremented)
      ÔööÔöÇ StockBaseDomain<T> Ôćĺ + @Column stockId, date        (foreign key + trading date)
          ÔöťÔöÇÔöÇ Stock              Ôćĺ table: stock               (exchange, code, name, status...)
          ÔöťÔöÇÔöÇ StockQuotes        Ôćĺ table: stock_quotes        (open, high, low, volume...)
          ÔöťÔöÇÔöÇ StockCandles       Ôćĺ table: stock_candles       (OHLCV data)
          ÔööÔöÇÔöÇ StockRankDaily     Ôćĺ table: stock_rank_daily    (daily ranking)
```

Only the concrete classes at the bottom have `@Table`.
The `@MappedSuperclass` levels above add fields that every concrete entity inherits.

==== Code Example

[source,java]
----
// Level 1: Every entity needs a creation timestamp
@MappedSuperclass
public abstract class MinBaseDomain<T extends MinBaseDomain<T>> {

    @Column
    public LocalDateTime createTime;   // Auto-set by the framework on INSERT
}

// Level 2: Add update tracking and optimistic locking
@MappedSuperclass
public abstract class BaseDomain<T extends BaseDomain<T>> extends MinBaseDomain<T> {

    @Column
    public LocalDateTime updateTime;   // Auto-refreshed on every INSERT / UPDATE

    @Column
    public int version;                 // Auto-incremented on UPDATE (optimistic lock)
}

// Level 3: Add fields shared by all stock-related entities
@MappedSuperclass
public abstract class StockBaseDomain<T extends StockBaseDomain<T>> extends BaseDomain<T> {

    @Column(notNull = true, updatable = false, comment = "stock id")
    public Long stockId;

    @Column(notNull = true, updatable = false, comment = "trading date")
    public LocalDate date;
}

// Concrete entity: inherits createTime/updateTime/version/stockId/date from the chain
@Table(name = "stock_quotes", comment = "daily stock quotes",
       indexes = @Index(name = "${DEFAULT_VALUE}", unique = true,
                        fieldList = {"stockId", "date"}))
public class StockQuotes extends StockBaseDomain<StockQuotes> {

    @Generator(value = "io.army.generator.snowflake.Snowflake8Generator",
               params = @Param(name = "startTime", value = "1779111192831"))
    @Column
    public long id;

    @Column(precision = 9, scale = 3, comment = "open price")
    public BigDecimal open;
    // ... more fields
}
----

==== Key Points

1. A `@MappedSuperclass` never maps to a table ÔÇö only `@Table` subclasses do.
2. All `@Column` annotations from every `@MappedSuperclass` in the chain are **inherited** by the concrete subclass.
3. Fields without `@Column` are **not persisted**.
4. You can nest `@MappedSuperclass` to build reusable inheritance chains.
5. For small projects, skip `@MappedSuperclass` and declare all fields directly ÔÇö it's entirely optional.

=== @Inheritance / @DiscriminatorValue ÔÇö Single-Table Inheritance

While `@MappedSuperclass` shares field definitions, `@Inheritance` implements the **single-table inheritance** strategy: a parent class and its subclasses all live in one database table, with a discriminator column telling them apart.

At startup, `++TableMetaUtils.discriminator++` locates the field named in `@Inheritance` and verifies it is an `Enum` type. `++TableMetaUtils.discriminatorValue++` reads `@DiscriminatorValue` on each child class and validates it against the enum constants.

[source,java]
----
// Parent: the root of the inheritance tree
@Table(name = "u_user", comment = "User entity")
@Inheritance("status")  // "status" field acts as the discriminator
public abstract class User extends BaseDomain<User> {

    @Column(comment = "User status")
    public UserStatus status;  // Must be an enum implementing CodeEnum
}

// Child: identified by discriminator value "ADMIN"
@DiscriminatorValue("ADMIN")
public class Admin extends User { }

// Child: identified by discriminator value "MEMBER"
@DiscriminatorValue("MEMBER")
public class Member extends User { }
----

The discriminator field must be an `Enum` implementing `io.army.struct.CodeEnum`.
The framework uses this column to filter rows by entity subtype in queries.

[IMPORTANT]
====
Only **one** `@Inheritance` is allowed per class hierarchy.
If multiple superclasses have `@Inheritance`, Army throws `MetaException`.
The parent table with `@Inheritance` cannot be marked as `immutable`.
====

TIP: `@Inheritance` and `@MappedSuperclass` serve different purposes.
Use `@MappedSuperclass` when you just want to share field definitions.
Use `@Inheritance` when you want polymorphic queries ÔÇö storing different entity types in one table and filtering by discriminator.

=== @DdlMode ÔÇö DDL Generation Control

Controls whether Army generates `CREATE TABLE` DDL during schema initialization.
At runtime, `++TableMetaUtils.tableCreateDdl++` reads `@Table.ddlMode()`:

[cols="1,4",options="header"]
|===
| Value | Behavior
| `CREATE` (default) | Always generate `CREATE TABLE` DDL
| `NONE` | Never generate DDL. Use for tables managed externally (e.g., by Flyway or Liquibase)
| `DEFAULT` | Check `META-INF/army/TableMeta.properties` key `{className}.Table.ddlMode`; fallback to `CREATE`
|===

[source,java]
----
// Prevent DDL generation for a legacy table
@Table(name = "legacy_data", comment = "Legacy data", ddlMode = DdlMode.NONE)
public class LegacyData { }
----

=== @Codec ÔÇö Reserved for Future Use

`@Codec` is a marker annotation with no current runtime effect.
The `++TableFieldMeta++` constructor records its presence in `CODEC_MAP`, but the framework does not act on it yet.
It is reserved as an extension point for future custom encode/decode interceptors.

=== TableMeta.properties ÔÇö Runtime Configuration

`TableMeta.properties` is the **runtime override hub** of Army.
Place this file at `META-INF/army/TableMeta.properties` on the classpath (typically +src/main/resources/META-INF/army/TableMeta.properties+).
Army scans **all JARs and directories** on the classpath and merges entries from every location ÔÇö this lets you change annotation attributes without recompiling.

==== Placeholder Expressions Reference

Many annotation attributes accept placeholder expressions instead of literal values.
These placeholders are resolved against `META-INF/army/TableMeta.properties` at startup.
They are defined as constants in `++TableMetaUtils++`:

[cols="2,3,5",options="header"]
|===
| Placeholder | Constant | Behavior
| `++${DEFAULT}++` | `TableMetaUtils.DEFAULT_EXP` | Check `META-INF/army/TableMeta.properties` first; if not found, use the framework's built-in convention (e.g., camelCaseÔćĺsnake_case for column names)
| `++${RUNTIME}++` | `TableMetaUtils.RUNTIME_EXP` | **Must** be in `META-INF/army/TableMeta.properties`; throws `MetaException` if missing. Use when the value depends on the deployment environment
| `++${OPTIONAL}++` | `TableMetaUtils.OPTIONAL_EXP` | Check `META-INF/army/TableMeta.properties` first; if not found, silently skip the element (used mainly for index types). Not all attributes support `++${OPTIONAL}++`
| `++${DEFAULT_VALUE}++` | `TableMetaUtils.DEFAULT_VALUE_EXP` | Skip `META-INF/army/TableMeta.properties` entirely; always use the convention-based default (e.g., auto-generated index names)
|===

[IMPORTANT]
====
`++${DEFAULT}++` and `++${DEFAULT_VALUE}++` are NOT the same:
* `++${DEFAULT}++` ÔÇö allows override via properties, falls back to convention if not found * `++${DEFAULT_VALUE}++` ÔÇö bypasses properties entirely, always uses the framework default Use `++${DEFAULT_VALUE}++` when you never want the value accidentally overridden.
====

==== What Happens When Not Configured

When you use a placeholder in your annotation, the framework enters a resolution chain.
Here is the **exact fallback behavior** for every attribute ÔÇö what happens when the property is **not configured** in `META-INF/army/TableMeta.properties`:

[cols="2,3,3",options="header"]
|===
| Attribute (Placeholder) | Annotation | Behavior When NOT Found

.2+^| **Table name**
`++${DEFAULT}++` | `@Table(name = "${DEFAULT}")` | Convert class simple name to snake_case: +`_MetaBridge.camelToLowerCase()`+
Ôćĺ `StockQuotes` Ôćĺ `"stock_quotes"`
Ôćĺ `UserAccount` Ôćĺ `"user_account"`
| `@Table(name = "${RUNTIME}")` | Throw `MetaException` ÔÇö no fallback

.2+^| **Table comment**
`++${DEFAULT}++` | `@Table(comment = "${DEFAULT}")` | Convert class simple name to spaced comment: +`_MetaBridge.camelToComment()`+
Ôćĺ `StockQuotes` Ôćĺ `"Stock Quotes"`
Ôćĺ `UserOrder` Ôćĺ `"User Order"`
| `@Table(comment = "${RUNTIME}")` | Throw `MetaException` ÔÇö no fallback

.2+^| **Catalog / Schema**
`++${DEFAULT}++` | `@Table(catalog = "${DEFAULT}")` | Empty string Ôćĺ **database default**
| `@Table(catalog = "${RUNTIME}")` | Throw `MetaException` ÔÇö no fallback

.2+^| **Column name**
`++${DEFAULT}++` | `@Column(name = "${DEFAULT}")` | Convert field name camelCaseÔćĺsnake_case: +`_MetaBridge.camelToLowerCase()`+

| `@Column(comment = "${RUNTIME}")` | Throw `MetaException` ÔÇö no fallback
| (empty string) | (no `comment` value) | Same as `++${DEFAULT}++` ÔÇö reserved Ôćĺ auto, business Ôćĺ error

.2+^| **Column precision**

| `RUNTIME_EXP` | Throw `MetaException` ÔÇö no fallback

.3+^| **Generator type**
`RUNTIME` | `@Generator(type = GeneratorType.RUNTIME)` | Throw `MetaException` ÔÇö no fallback; property **must** provide `Generator.type` or `GeneratorStrategy`

`++${DEFAULT_VALUE}++` | `@Index(name = "${DEFAULT_VALUE}")` | Always auto-generated (ignores properties):

| `++${DEFAULT}++` | `@Index(name = "${DEFAULT}")` | Falls back to `++${DEFAULT_VALUE}++` behavior (auto-generated name)
| `++${OPTIONAL}++` | `@Index(name = "${OPTIONAL}")` | `null` Ôćĺ **index is silently skipped**

.3+^| **Index type**
`++${DEFAULT}++` | `@Index(type = "${DEFAULT}")` | Empty string Ôćĺ **database default** index type
| `++${RUNTIME}++` | `@Index(type = "${RUNTIME}")` | Throw `MetaException` ÔÇö no fallback

|===

==== Usage Examples

[source,java]
----
// ÔöÇÔöÇ Table name: auto-derive from class name, allow override via properties ÔöÇÔöÇ
// Without properties: StockQuotes Ôćĺ "stock_quotes"
// With properties: com.example.domain.StockQuotes.Table.name=stock_v2
@Table(name = "${DEFAULT}", comment = "stock data")

// ÔöÇÔöÇ Table name: MUST be in properties, startup fails otherwise ÔöÇÔöÇ
// Used when team A defines the entity but team B controls deployment name
@Table(name = "${RUNTIME}", comment = "cross-team table")

// ÔöÇÔöÇ Column name: camelCase Ôćĺ snake_case by default, configurable ÔöÇÔöÇ
// Without properties: userName Ôćĺ "user_name"
// With properties: com.example.Stock.userName.Column.name=login_name
@Column(name = "${DEFAULT}")
public String userName;

// ÔöÇÔöÇ Column precision: must be configured at startup ÔÇö varies by deployment ÔöÇÔöÇ
// Without properties: throws MetaException
// With properties: com.example.Stock.embedding.Column.precision=768
@Column(precision = Column.RUNTIME_EXP)
public String embedding;

// ÔöÇÔöÇ Index name: always auto-generate, never override ÔöÇÔöÇ
// @Table(name = "stock") + @Index(fieldList = "userId")
// Ôćĺ unique: uni_stock_user_id, non-unique: idx_stock_user_id
@Index(name = "${DEFAULT_VALUE}", fieldList = "userId")

// ÔöÇÔöÇ Index name: default generation but allow override via properties ÔöÇÔöÇ
// Without properties: same as ${DEFAULT_VALUE}
// With properties: com.example.IndexMeta[0].name=my_index
@Index(name = "${DEFAULT}", fieldList = "userId")

// ÔöÇÔöÇ Optional index: skip entirely on staging, keep in production ÔöÇÔöÇ
// Without properties: index is silently skipped
// With properties: com.example.IndexMeta[0].type=btree Ôćĺ index is created
@Index(name = "idx_stock", type = "${OPTIONAL}", fieldList = "userId")

// ÔöÇÔöÇ Column default value: type-appropriate zero ÔöÇÔöÇ
// LocalDateTime Ôćĺ '1970-01-01 00:00:00'
// BigDecimal Ôćĺ 0
// String Ôćĺ ''
@Column(defaultValue = "${DEFAULT}", comment = "upload complete time")
public LocalDateTime uploadCompleteTime;

// ÔöÇÔöÇ Generator type: defer to deployment config ÔöÇÔöÇ
// Development: POST (auto-increment), Production: PRECEDE (Snowflake)
@Generator(type = GeneratorType.RUNTIME)
@Column
public long id;

| `GeneratorStrategy` | `com.example.domain.Stock.id.GeneratorStrategy` | `io.army.generator.Snowflake8GeneratorStrategy:{"startTime":1779111192831}`
| `Mapping.value` | `com.example.domain.Stock.id.Mapping.value` | `io.army.mapping.SqlBigIntType`
| `IndexMeta[n].name` | `com.example.domain.Stock.IndexMeta[0].name` | `idx_stock_custom`
| `IndexMeta[n].type` | `com.example.domain.Stock.IndexMeta[0].type` | `btree`
| `IndexMeta[n].field.X.collation` | `com.example.domain.Stock.IndexMeta[0].field.code.collation` | `en_US.utf8`
| `IndexMeta[n].field.X.opclass` | `com.example.domain.Stock.IndexMeta[0].field.embedding.opclass` | `vector_cosine_ops`
|===

[IMPORTANT]
====
Property keys are **case-sensitive**.
Keyword components ÔÇö `Table`, `Column`, `Generator`, `GeneratorStrategy`, `Mapping`, `IndexMeta` ÔÇö must appear **exactly** with that capitalization.

The `IndexMeta[n]` syntax uses a 0-based index: `[0]` refers to the **first** `@Index` in `@Table.indexes` array, `[1]` to the second, etc.
The `field.{fieldName}` segment uses the **Java field name** (not the column name).
====

NOTE: The `GeneratorStrategy` value format is `fully.qualified.StrategyClass[:jsonParams]`.
The optional JSON parameters after the colon are parsed by the strategy's static `create(String)` factory method.

== MappingType ÔÇö Java-to-Database Type Mapping

Every domain field in Army must declare its *type mapping* ÔÇö the rule that converts between a Java type and a database column type.
This mapping is defined either implicitly (via default type inference) or explicitly (via the `@Mapping` annotation).
The `MappingType` interface is the core abstraction that orchestrates this conversion.

=== Overview: MappingType Interface

[source,java]
----
public sealed interface MappingType permits AbstractMappingType, ArrayMappingType {

    /// The Java type that this mapping represents.
    Class<?> javaType();

    /// Resolve to the concrete SQL data type for the current database dialect.
    DataType map(ServerMeta meta);

    /// Convert a Java value Ôćĺ SQL parameter value (before binding).
    Object beforeBind(DataType dataType, MappingEnv env, Object source);

    /// Convert a database value Ôćĺ Java object (after fetching).
    Object afterGet(DataType dataType, MappingEnv env, Object source);

    /// Return the corresponding array MappingType for this scalar type.
    MappingType arrayTypeOfThis();
}
----

A `MappingType` implementation defines four things:

. **Which Java type** it handles ÔÇö returned by `javaType()`.
. **Which SQL type** it maps to ÔÇö resolved by `map(ServerMeta)`, which is database-dialect-aware.
. **How values are serialized** before binding to a prepared statement ÔÇö `beforeBind()`.
. **How values are deserialized** after fetching from a `ResultSet` ÔÇö `afterGet()`.

For example, `io.army.mapping.StringType` maps `java.lang.String` Ôćĺ MySQL `VARCHAR` / PostgreSQL `VARCHAR` / Oracle `VARCHAR2`, and during `beforeBind()` it can convert `Number`, `Boolean`, `Enum`, `LocalDateTime`, etc. into their string representations automatically.

=== Default Type Inference

When a domain field has *no* `@Mapping` annotation, Army infers the `MappingType` from the field's Java type via `io.army.mapping._MappingFactory.getDefaultIfMatch()`.

==== Enum Type Inference

When a domain field's Java type is an `enum`, Army has three distinct mapping strategies:

[cols="1,2,3,3",options="header"]
|===
| Strategy | Implements/Extends | Stored in DB as | Use Case

| `NameEnumType`
| (plain `enum`, no marker interface)
| `VARCHAR` (or PostgreSQL native `ENUM` with `@DefinedType`)
| Simple enums where `name()` is human-readable enough (e.g., `NORMAL`, `DELISTED`)

| `TextEnumType`
| `implements TextEnum`
| `VARCHAR` (or PostgreSQL native `ENUM` with `@DefinedType`)
| Enum values need a display text different from the Java constant name (e.g., `ZORO` Ôćĺ `"Roronoa Zoro"`)

| `CodeEnumType`
| `implements CodeEnum`
| `INTEGER`
| Fixed, stable integer codes for maximum storage efficiency and ABI compatibility
|===

===== NameEnumType ÔÇö Default Enum Strategy

A plain `enum` with no marker interface uses `NameEnumType`.
The stored value is `Enum.name()`, which is the Java constant identifier.

[source,java]
----
public enum StockStatus {
    NORMAL,     // Ôćĺ stored as "NORMAL"
    DELISTED,   // Ôćĺ stored as "DELISTED"
    STOPT,      // Ôćĺ stored as "STOPT"
    UNKNOWN     // Ôćĺ stored as "UNKNOWN"
}

@Column(notNull = true, defaultValue = "'NORMAL'", comment = "ŔéíšąĘňťĘňŞéňť║šŐÂŠÇü")
public StockStatus status;

// Default inference: StockStatus Ôćĺ NameEnumType Ôćĺ VARCHAR column
----

Under the hood, `NameEnumType.beforeBind()` calls `((Enum<?>) source).name()`, and `afterGet()` calls `Enum.valueOf(enumClass, string)`.
The mapping to SQL types is dialect-dependent:

- **MySQL**: native `ENUM('NORMAL','DELISTED','STOPT','UNKNOWN')`
- **PostgreSQL**: `VARCHAR` by default; becomes a native `CREATE TYPE ... AS ENUM` if the enum class is annotated with `@DefinedType`
- **H2**: native `ENUM`
- **SQLite**: `VARCHAR`

TIP: A plain enum mapped via `NameEnumType` is safe for adding new constants at the end of the list, but *renaming* or *reordering* existing constants will break existing data (the stored string won't match).

===== TextEnumType ÔÇö Text-Based Enum with Custom Label

`NameEnumType` stores `Enum.name()` ÔÇö but `name()` is constrained to valid Java identifiers.
What if you need the database to store a more descriptive text, like `"Roronoa Zoro"` instead of `"ZORO"`?

That's what `TextEnumType` is for.
By implementing `io.army.struct.TextEnum`, the enum provides a `text()` method that returns the value to be stored:

[source,java]
----
public enum QinArmy implements TextEnum {

    ZORO("Roronoa Zoro"),         // stored as "Roronoa Zoro"
    ANZAI("Mitsuyoshi Anzai");    // stored as "Mitsuyoshi Anzai"

    private final String text;

    QinArmy(String text) {
        this.text = text;
    }

    @Override
    public String text() {
        return this.text;
    }
}

@Column(comment = "ňëĹňúźňÉŹšž░")
public QinArmy swordsman;  // Ôćĺ TextEnumType, stored as VARCHAR
----

`TextEnumType.beforeBind()` calls `((TextEnum) source).text()` instead of `name()`, and `afterGet()` reads the stored text and looks up the enum instance via an internal `Map<String, TextEnum>` built at initialization time.

TextEnumType vs NameEnumType:

[cols="2,3,3",options="header"]
|===
| Aspect | `NameEnumType` | `TextEnumType`

| Stored value source
| `Enum.name()` (Java constant identifier)
| `TextEnum.text()` (arbitrary string)

| Rename safety
| Renaming the Java constant breaks data
| Renaming the Java constant is safe ÔÇö only `text()` matters

| Database readability
| Stored as Java-style identifiers (`NORMAL`)
| Stored as human-readable text (`"Roronoa Zoro"`)

| Special enum support
| Handles `Month` and `DayOfWeek` (see SpecialEnums)
| No special enum handling

| `@DefinedType` support
| Ôťô (PostgreSQL native `ENUM`)
| Ôťô (PostgreSQL native `ENUM`)
|===

===== CodeEnumType ÔÇö Integer-Code Enum

When the enum values have stable, semantically-meaningful integer codes, `CodeEnumType` is the best choice.
It stores only the integer, which is the most storage-efficient option and is immune to refactoring (the code stays unchanged even if the Java constant is renamed).

[source,java]
----
public enum BankAccountType implements CodeEnum {

    BANK(0),
    LENDER(100),
    BORROWER(200),
    PARTNER(300),
    LENDER_BUSINESS(400),
    BORROWER_BUSINESS(500),
    GUARANTOR(600);

    private final short code;

    BankAccountType(short code) {
        this.code = code;
    }

    @Override
    public final int code() {
        return this.code;
    }
}

@Column(comment = "Ŕ┤ŽŠłĚš▒╗ň×ő")
public BankAccountType accountType;  // Ôćĺ CodeEnumType, stored as INT
----

`CodeEnumType.beforeBind()` returns `((CodeEnum) source).code()` as an `Integer`. `afterGet()` is flexible ÔÇö it can deserialize from `Integer`, `Long`, `Short`, `Byte`, `BigInteger`, or `String`.

CodeEnumType is mapped to an `INTEGER` column on all dialects.
It is the most robust strategy for long-lived enums whose semantics are encoded in numbers (e.g., protocol codes, status codes, wire-format constants).

IMPORTANT: Never use consecutive integer codes (`0, 1, 2, 3, ...`) for `CodeEnum`.
Always leave gaps between codes ÔÇö use increments of **10** (e.g., `0, 10, 20, 30`) or **100** (e.g., `0, 100, 200, 300`).
Consecutive values make it impossible to insert a new constant between existing ones without renumbering, which would corrupt existing data.
With spaced codes, you can always insert a new value ÔÇö e.g., between `LENDER(100)` and `BORROWER(200)` you can add `LENDER_PREMIUM(150)`.

WARNING: Never implement both `CodeEnum` and `TextEnum` on the same enum class.
Army will reject it at startup with an `IllegalArgumentException`.

===== PostgreSQL Native ENUM via @DefinedType

Both `NameEnumType` and `TextEnumType` support PostgreSQL native `ENUM` types through the `io.army.struct.DefinedType` annotation.
When applied to the enum class, Army generates a `CREATE TYPE` DDL and maps the field to the named PostgreSQL enum type instead of `VARCHAR`:

[source,java]
----
@DefinedType(name = "stock_status_enum", category = TypeCategory.ENUM)
public enum StockStatus {
    NORMAL,
    DELISTED,
    STOPT,
    UNKNOWN
}

@Column(comment = "ŔéíšąĘňťĘňŞéšŐÂŠÇü")
public StockStatus status;

// Without @DefinedType Ôćĺ VARCHAR
// With @DefinedType(name = "stock_status_enum") Ôćĺ PostgreSQL: CREATE TYPE stock_status_enum AS ENUM (...)
----

This is especially useful when the enum type needs to be shared across multiple tables, or when you want the database to enforce value constraints at the schema level.

===== Special Enums: Month and DayOfWeek

`NameEnumType` has built-in intelligent handling for `java.time.Month` and `java.time.DayOfWeek`.
These enums are commonly returned as integers by database functions (e.g., `MONTH()`, `DAYOFWEEK()`) and as strings by others (`MONTHNAME()`, `DAYNAME()`).

**Month** ÔÇö `NameEnumType.afterGet()` accepts:

- `Integer` (1ÔÇô12) Ôćĺ `Month.of(n)` ÔÇö covers MySQL `MONTH()` / PostgreSQL `EXTRACT(MONTH FROM ...)`
- `Long` (1ÔÇô12) Ôćĺ `Month.of((int) n)`
- `String` (`"JANUARY"`, `"FEBRUARY"`, ...) Ôćĺ `Month.valueOf(name.toUpperCase())` ÔÇö covers `MONTHNAME()`
- `LocalDate`, `YearMonth`, `MonthDay`, `LocalDateTime`, `OffsetDateTime`, `ZonedDateTime` Ôćĺ `Month.from(temporal)`

[source,java]
----
// MySQL: SELECT MONTH(create_time) m, MONTHNAME(create_time) mn FROM ...
// Both m (integer) and mn (string) can be read into a Month field
@Column
public Month birthMonth;  // Ôćĺ NameEnumType + Month special handling
----

**DayOfWeek** ÔÇö `NameEnumType.afterGet()` accepts:

- `Integer` ÔÇö on MySQL, `1` maps to `SUNDAY` (MySQL `DAYOFWEEK()` convention); otherwise `DayOfWeek.of(n)`
- `Long` Ôćĺ same as integer
- `String` (`"MONDAY"`, `"TUESDAY"`, ...) Ôćĺ `DayOfWeek.valueOf(name.toUpperCase())`
- `LocalDate`, `LocalDateTime`, `OffsetDateTime`, `ZonedDateTime` Ôćĺ `DayOfWeek.from(temporal)`

[source,java]
----
// MySQL: SELECT DAYOFWEEK(create_time) dow, DAYNAME(create_time) dn FROM ...
// In MySQL, DAYOFWEEK returns 1=Sunday, 2=Monday, ..., 7=Saturday
// NameEnumType handles this MySQL-specific mapping automatically
@Column
public DayOfWeek tradingDay;  // Ôćĺ NameEnumType + DayOfWeek special handling
----

=== @Mapping Annotation: Explicit Mapping Control

When the default inference is insufficient ÔÇö or when you need a non-standard database type ÔÇö use the `io.army.annotation.Mapping` annotation.

==== Annotation Definition

[source,java]
----
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Mapping {

    /// Fully-qualified class name of the MappingType implementation.
    /// Used as a string to avoid compile-time dependencies on dialect-specific classes.
    String value() default "";

    /// The MappingType class itself (takes precedence over value()).
    Class<?> type() default void.class;

    /// Extra configuration parameters (e.g., PostgreSQL enum type name).
    String[] params() default {};

    /// Java method reference for constructing or creating instances of the mapped type.
    /// Format: "ClassName::new" (constructor) or "ClassName::methodName" (static factory).
    /// Consumed by PgSingleRangeType.fromMethod / PgMultiRangeType.fromMethod and array variants
    /// via PgRangeType.createRangeFunction; resolved by _ArmyPgRangeType.loadConstructor / loadFactoryMethod.
    /// For non-range types, dispatched via _MappingFactory.doMap.
    String func() default "";

    /// Character set for text-based binary mappings (e.g., LongBlobType + InputStream).
    String charset() default "";

    /// Element types for collection mappings (e.g., MySQL SET).
    Class<?>[] elements() default {};
}
----

==== Resolution Priority

The `@Mapping` value follows the same placeholder mechanism as other annotations:

. If `type()` is set (not `void.class`), it takes precedence over `value()`.
. If `value()` is `"${DEFAULT}"`, Army checks `TableMeta.properties`; if not configured, falls back to default type inference.
. If `value()` is `"${RUNTIME}"`, the property *must* be configured in `TableMeta.properties` or a `MetaException` is thrown.
. Otherwise, `value()` is a fully-qualified class name resolved via `Class.forName()`.

| Maps to `JSONB` on PostgreSQL, falls back to `JSON` on MySQL

| `JsonbType`
| `String`
| Strictly maps to `JSONB` (PostgreSQL-only, throws on other dialects)

| `ArmyJsonType`
| (abstract base)
| Base class for all JSON types; provides auto-expansion of Map/List generics
|===

`PreferredJsonbType` is the most commonly used because it provides optimal storage and indexing (`jsonb`) on PostgreSQL while gracefully degrading to `json` on MySQL.

=== How TypeMappingHandler Resolves to SQL Types

After Army determines a field's `MappingType`, the `io.army.dialect.TypeMappingHandler` interface finalizes the concrete `DataType` for the target database:

[source,java]
----
public sealed interface TypeMappingHandler permits TypeMappingHandlerSupport {

    /// For a given custom type name and MappingType array, return the concrete DataType.
    DataType apply(String typeName, MappingType[] typeArray, int index);
}
----

The implementation `io.army.dialect.TypeMappingHandlerSupport` performs this resolution:

. **Custom type lookup** ÔÇö First checks if a `DefinedTypeMapFunc` is registered (for user-defined database types like PostgreEnum).
. **Named type lookup** ÔÇö Checks the dialect's internal type registry (e.g., `"vector"` Ôćĺ `PgType.VECTOR`).
. **Array detection** ÔÇö If the type name contains `[` (e.g., `"integer[]"`), it resolves the element type then wraps it via `arrayTypeOfThis()`.
. **Fallback** ÔÇö If no match is found, returns the `defaultType` (the `MappingType.map()` result).

[source,java]
----
// Simplified resolution logic in TypeMappingHandlerSupport
final MappingType handleDefinedType(final String typeName, final MappingType defaultType) {
    // 1. Try user-defined type function
    MappingType type = func.apply(typeName, serverMeta, typeMap);
    if (type != null) return type;

    // 2. Check if it's an array type (e.g., "integer[]")
    final int index = typeName.indexOf('[');
    if (index < 0) {
        type = this.typeMap.getOrDefault(typeName, defaultType);
    } else {
        type = this.typeMap.get(typeName.substring(0, index));  // element type
        if (type != null) {
            type = ((MappingType.SqlArray) type.arrayTypeOfThis()).unlimited();
        } else {
            type = defaultType;
        }
    }
    return type;
}
----

This design means the `MappingType` hierarchy is database-agnostic ÔÇö each `MappingType` declares its target `DataType` per-dialect in `map()`, and `TypeMappingHandler` provides the final resolution for custom/composite/array types.
In most cases, developers interact only with the `@Mapping` annotation; the handler chain operates transparently behind the scenes.

=== Custom MappingType ÔÇö Extending the Type System

When the built-in mapping types don't fit your domain, Army provides two abstract base classes for creating custom types.

==== Step 1: Choose the Right Base Class

[cols="2,5,3",options="header"]
|===
| Scenario | Base Class | Example

| Non-array Java type
| Extend `io.army.mapping.UserMappingType`
| A custom `Currency` type mapped to `NUMERIC(10,2)`

| Array Java type
| Extend `io.army.mapping.UserArrayType`
| A custom `IntRange[]` type mapped to PostgreSQL `INT4RANGE[]`
|===

`UserMappingType` extends `AbstractMappingType`, which already provides `beforeBind()` / `afterGet()` defaults (pass-through to `String`).
`UserArrayType` extends `UserMappingType` and additionally implements `ArrayMappingType`.

==== Step 2: Implement the Corresponding `Sql*` Marker Interface

Every mapping type must declare what *kind* of SQL value it represents by implementing one of `MappingType`'s inner marker interfaces.
The Army framework uses these markers for validation and type compatibility checks.

Common marker interfaces:

[cols="2,3,4",options="header"]
|===
| Category | Interface | When to Use

| Number
| `MappingType.SqlInteger`
| Maps to `INT`. Your type handles integer values.

| Number
| `MappingType.SqlDecimal`
| Maps to `NUMERIC`. Your type handles decimal values.

| String
| `MappingType.SqlText`
| Maps to `TEXT`. Your type handles long unbounded strings.

| String
| `MappingType.SqlString`
| Maps to `VARCHAR`. Your type handles short bounded strings.

| Binary
| `MappingType.SqlBlob`
| Maps to `BYTEA` / `BLOB`. Your type handles binary data.

| JSON
| `MappingType.SqlJson`
| Maps to `JSON`. Your type stores JSON documents.

| Extension
| `MappingType.SqlVector`
| Maps to pgvector `VECTOR` type. Combine with `MappingType.SqlExtension` to auto-create the extension.

| Time
| `MappingType.SqlLocalDateTime`
| Maps to `TIMESTAMP`. Your type handles date-time values.
|===

[TIP]
====
The full list of marker interfaces is in `io.army.mapping.MappingType`:
`SqlBoolean`, `SqlInteger`, `SqlFloat`, `SqlDecimal`, `SqlText`, `SqlString`, `SqlBlob`, `SqlBinary`, `SqlBit`, `SqlJson`, `SqlJsonb`, `SqlLocalDate`, `SqlLocalTime`, `SqlLocalDateTime`, `SqlOffsetDateTime`, `SqlDuration`, `SqlVector`, `SqlExtension`, etc.

Pick the one that most closely describes your target SQL type ÔÇö each has a single unambiguous semantic meaning.
====

==== Step 3: Follow the Implementation Pattern

All built-in mapping types follow a consistent pattern.
Your custom type must adhere to it for framework compatibility:

[source,java]
----
/// Custom mapping: maps a Java `Currency` record to PostgreSQL `NUMERIC(10,2)`.
public final class CurrencyType extends UserMappingType
        implements MappingType.SqlDecimal {

    // 1. Static factory method ÔÇö required for @Mapping resolution
    public static CurrencyType from(final Class<?> javaType) {
        if (javaType != Currency.class) {
            throw errorJavaType(CurrencyType.class, javaType);
        }
        return INSTANCE;
    }

    public static final CurrencyType INSTANCE = new CurrencyType();

    // 2. Constructor must be non-public (private or package-private)
    private CurrencyType() {
    }

    // 3. Declare the Java type this mapping handles
    @Override
    public Class<?> javaType() {
        return Currency.class;
    }

    // 4. Map to the concrete SQL DataType per database
    @Override
    public DataType map(final ServerMeta meta) {
        switch (meta.serverDatabase()) {
            case PostgreSQL:  return PgType.NUMERIC;
            case MySQL:       return MySQLType.DECIMAL;
            default:          throw mapError(this, meta);
        }
    }

    // 5. Serialize Java value Ôćĺ SQL parameter
    @Override
    public String beforeBind(DataType dataType, MappingEnv env, Object source) {
        if (!(source instanceof Currency c)) {
            throw paramError(this, dataType, source, null);
        }
        return c.value().toString();  // e.g., "99.99"
    }

    // 6. Deserialize database value Ôćĺ Java object
    @Override
    public Currency afterGet(DataType dataType, MappingEnv env, Object source) {
        if (source instanceof Currency c) {
            return c;
        }
        if (source instanceof BigDecimal bd) {
            return new Currency(bd);
        }
        if (source instanceof String s) {
            return new Currency(new BigDecimal(s));
        }
        throw dataAccessError(this, dataType, source, null);
    }

    // 7. Required: override hashCode/equals for singleton types
    @Override
    public int hashCode() {
        return System.identityHashCode(this);
    }

    @Override
    public boolean equals(Object obj) {
        return obj instanceof CurrencyType;
    }
}
----

[IMPORTANT]
====
The `from(Class)` static factory method is how `_MappingFactory.doMap` reflectively creates your MappingType.
Without it, the framework cannot instantiate your type.
The constructor must be non-public ÔÇö all built-in types use `private` constructors and expose a singleton `INSTANCE`.
====

==== Array Type Example

For array types, extend `UserArrayType` and implement `MappingType.SqlArray`:

[source,java]
----
/// Custom mapping: maps `Currency[]` to PostgreSQL `NUMERIC[]`.
public final class CurrencyArrayType extends UserArrayType
        implements MappingType.SqlArray {

    public static CurrencyArrayType from(final Class<?> javaType) {
        if (javaType != Currency[].class) {
            throw errorJavaType(CurrencyArrayType.class, javaType);
        }
        return INSTANCE;
    }

    public static final CurrencyArrayType INSTANCE = new CurrencyArrayType();

    private CurrencyArrayType() {
    }

    @Override
    public Class<?> javaType() {
        return Currency[].class;
    }

    @Override
    public Class<?> underlyingJavaType() {
        return Currency.class;
    }

    @Override
    public MappingType elementType() {
        return CurrencyType.INSTANCE;
    }

    @Override
    public DataType map(final ServerMeta meta) {
        switch (meta.serverDatabase()) {
            case PostgreSQL:  return PgType.NUMERIC_ARRAY;
            case MySQL:       throw mapError(this, meta);  // MySQL doesn't support arrays
            default:          throw mapError(this, meta);
        }
    }

    // beforeBind / afterGet omitted for brevity ÔÇö see built-in array types for reference

    @Override
    public int hashCode() {
        return System.identityHashCode(this);
    }

    @Override
    public boolean equals(Object obj) {
        return obj instanceof CurrencyArrayType;
    }
}
----

[TIP]
====
For array types, the element type is obtained from the field's component type via reflection ÔÇö you do NOT need `@Mapping(elements = ...)`.
The `elements` attribute is only for generic collections (`List<Foo>`, `Set<Bar>`) where type erasure loses the element type.

Before writing your custom type, verify:

- [ ] Non-array type Ôćĺ extends `UserMappingType`; Array type Ôćĺ extends `UserArrayType`

- [ ] Provides a `public static ... from(Class)` factory method

- [ ] Overrides `javaType()`, `map()`, `beforeBind()`, `afterGet()`

[cols="2,5,4",options="header"]
|===
| Stage | What Happens | Example

| 1. **Annotation Processing**
| Army reads `@Mapping` (or infers default) to determine `MappingType`
| `@Mapping("io.army.mapping.optional.VectorType")` Ôćĺ `VectorType`

| 2. **Dialect Resolution**
| `MappingType.map(ServerMeta)` returns database-specific `DataType`

| 3. **TypeHandler Finalization**

| 5. **Value Fetching**
| `MappingType.afterGet()` deserializes database value to Java object
| `VectorType.afterGet()`: `"[0.1,0.2]"` Ôćĺ `float[]{0.1,0.2}`

=== Architectural Layers

```
Presentation Layer (Criteria API)
  Ôćô Generates Statement tree
Parsing Layer (Dialect Parser)
  Ôćô Dialect-specific SQL + parameter binding
Metadata Layer (Meta)
  Ôćô Table / field / index metadata queries
Execution Layer (Executor)
  Ôćô JDBC driver
Session Layer (Session)
  Ôćô Transaction context, lifecycle management
Type Mapping Layer (Mapping)
  Ôćô Java Ôćö SQL bidirectional type conversion
```

==== Presentation Layer (Criteria API)

`SQLs` (standard), `MySQLs` (MySQL dialect), `Postgres` (PostgreSQL dialect) ÔÇö builds standardized Statement trees using a type-safe DSL.

==== Parsing Layer (Dialect Parser)

Dialect parsers (`ArmyParser` / `MySQLParser` / `PostgreParser`) convert Statement trees into native SQL for each database and bind parameters.

==== Metadata Layer (Meta)

* `TableMeta` ÔÇö Table metadata (table name, schema, field chain, primary key fields, discriminator fields)
* `FieldMeta` ÔÇö Field metadata (Java type, mapping type, generator metadata)
* `PrimaryFieldMeta` ÔÇö Primary key field metadata
* `IndexMeta` ÔÇö Index metadata (index name, field list, uniqueness)
* `ServerMeta` ÔÇö Server metadata (database type, version, dialect information)
* `SchemaMeta` ÔÇö Schema metadata registry

==== Execution Layer (Executor)

The executor interface provides JDBC-based result set processing and batch operations.

==== Session Layer (Session)

The `Session` / `SessionFactory` interfaces manage transactions, session context, and lifecycle.

== Advanced Features

=== Reserved Field Mechanism

Army reserves the following field names with framework-managed behavior:

[cols="1,1,3",options="header"]
|===
| Field | Type | Behavior
| `id` | Numeric | Primary key; automatic NOT NULL; non-updatable
| `createTime` | Temporal | Auto-populated on creation; non-updatable
| `updateTime` | Temporal | Auto-populated on update
| `version` | Integer | Optimistic lock version; auto-incremented on update
| `visible` | Boolean | Soft-delete flag; auto-filtered
|===

=== Soft Deletes

Use the `visible` field to implement logical deletion.
Rows marked as invisible are automatically filtered out in queries.

[source,java]
----
@Column
private Boolean visible;
----

Visibility mode configuration: `SUPPORT` (enable filtering), `NEVER` (no filtering), `WHITE_LIST` (whitelist mode).

=== Optimistic Locking

Use the `version` field for optimistic concurrency control: UPDATE statements automatically validate the version in the WHERE clause; the update fails if the version does not match.

[source,java]
----
@Column
private Integer version;
----

=== Inheritance Mapping in Depth

Army uses the single-table inheritance strategy: the parent and all child classes share one database table, with the discriminator column (`@Inheritance`) distinguishing which subtype each row belongs to.

Child classes use `@DiscriminatorValue` to specify their own enum value.
The framework automatically filters by the discriminator value in queries.

== Best Practices

=== Domain Class Design

1. Use generics to support inheritance mapping
2. Add comments to all fields
3. Use reserved fields judiciously (`id`, `createTime`, `updateTime`, `version`, `visible`)
4. Configure generators correctly (`PRECEDE` vs `POST`)
5. Externalize environment-specific configuration through `TableMeta.properties`

=== Criteria API Usage

1. Prefer the standard Criteria API (`SQLs`)
2. Use dialect APIs (`MySQLs` / `Postgres`) only when database-specific features are needed
3. Choose parameter binding (`param`) vs literal binding (`literal`) wisely
4. Be aware of the performance implications of CTEs and subqueries

=== Transaction Management

1. Use declarative transactions (`@Transactional`)
2. Set appropriate transaction isolation levels
3. Be mindful of the performance impact of long-running transactions
4. Use savepoints for fine-grained control

=== Performance Optimization

1. Use batch operations instead of looping single-row operations
2. Use indexes wisely (create them during DDL generation via `@Index`)
3. Avoid N+1 query problems
4. Use appropriate parameter binding strategies

=== Type Mapping

1. Prefer built-in type mappings
2. Use custom type mappings with caution
3. Configure JSON/XML codecs correctly
4. Pay attention to timezone handling

== FAQ

=== Metamodel Classes Not Generated by the Annotation Processor

Check: 1. Whether `annotationProcessors` is configured correctly 2. Whether the domain class uses the `@Table` annotation 3. Whether the compile classpath includes the `army-annotation` dependency

=== Snowflake Generator Duplicate startTime Warning

Issue: Using the same `startTime` parameter across multiple domain classes risks ID collisions.

Solution: 1. Use distinct `startTime` values for different entities 2. Suppress the warning with `-Aarmy.snowflakeStartTimeWarning=false`

=== Transaction Exception (IllegalTransactionStateException)

Solution: 1. Verify that a transaction has been properly started 2. Check the transaction propagation behavior configuration 3. Ensure database operations are executed within a transaction context

=== Type Mapping Errors

Solution: 1. Check whether the field type matches the database type 2. Configure the correct `MappingType` 3. Verify the JSON/XML codec configuration
